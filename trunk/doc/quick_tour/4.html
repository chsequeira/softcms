4. Архитектура
Вы мой герой! Кто бы мог подумать, что вы все еще будете здесь после первых трех глав? Вскоре ваши усилия будут вознаграждены. В предыдущих частях я не особенно углублялся в архитектуру фреймворка. Но поскольку именно она выделяет Symfony2 из толпы прочих фреймворков, давайте подробнее познакомимся с ней сейчас.

4.1.Структура папок
Структура папок в приложениях Symfony2 весьма гибкая, но структура папок в «песочнице» отражает отражает типичную и рекомендованную структуру приложения Symfony2:

app/: этот каталог содержит настройки приложения
src/: весь PHP код хранится в этой папке
web/: а эта папка должна быть вашей веб-директорией

4.1.1.Веб-директория
Корневая веб-директория — дом для всех общедоступных и статических файлов, таких как изображения, стили и файлы JavaScript. А еще там живут фронт-контроллеры:

  1. // web/app.php
  2. require_once __DIR__.'/../app/AppKernel.php';
  3.  
  4. use Symfony\Component\HttpFoundation\Request;
  5.  
  6. $kernel = new AppKernel('prod', false);
  7. $kernel->handle(new Request())->send();


Как и любой другой веб-контроллер, app.php использует класс ядра, AppKernel, для начальной загрузки приложений.



4.1.2. Директория приложений
Класс AppKernel — главная точка входа в настройки приложения и, как следствие, находится в директории app/.
Этот класс должен реализовывать четыре метода:

registerRootDir(): вовзращает настройки корневой директории
registerBundles(): возвращает массив всех связок, необходимых для запуска приложения (обратите внимание на ссылку на Application\HelloBundle\HelloBundle)
registerBundleDirs(): возвращает массив ассоциативных заменяемых значений и их домашних директорий
registerContainerConfiguration(): возвращает основную конфигурацию объекта (подробнее об этом — чуть позже).

Присмотритесь к реализации этих методов по умолчанию для лучшего понимания гибкости фреймворка.
Чтобы все эти штуки работали вместе, ядру требуется один файл папки src/:

  1. // app/AppKernel.php
  2. require_once __DIR__.'/../src/autoload.php';


4.1.3.Исходный каталог
Файл src/autoload.php ответственен за автозагрузку всех файлов, хранящихся в папке src/:

  1. // src/autoload.php
  2. $vendorDir = __DIR__.'/vendor';
  3.  
  4. require_once $vendorDir.'/symfony/src/Symfony/Component/HttpFoundation/UniversalClassLoader.php';
  5.  
  6. use Symfony\Component\HttpFoundation\UniversalClassLoader;
  7.  
  8. $loader = new UniversalClassLoader();
  9. $loader->registerNamespaces(array(
  10.     'Symfony'                        => $vendorDir.'/symfony/src',
  11.     'Application'                    => __DIR__,
  12.     'Bundle'                         => __DIR__,
  13.     'Doctrine\\Common\\DataFixtures' => $vendorDir.'/doctrine-data-fixtures/lib',
  14.     'Doctrine\\Common'               => $vendorDir.'/doctrine-common/lib',
  15.     'Doctrine\\DBAL\\Migrations'     => $vendorDir.'/doctrine-migrations/lib',
  16.     'Doctrine\\ODM\\MongoDB'         => $vendorDir.'/doctrine-mongodb/lib',
  17.     'Doctrine\\DBAL'                 => $vendorDir.'/doctrine-dbal/lib',
  18.     'Doctrine'                       => $vendorDir.'/doctrine/lib',
  19.     'Zend'                           => $vendorDir.'/zend/library',
  20. ));
  21. $loader->registerPrefixes(array(
  22.     'Swift_' => $vendorDir.'/swiftmailer/lib/classes',
  23.     'Twig_'  => $vendorDir.'/twig/lib',
  24. ));
  25. $loader->register();

UniversalClassLoader из Symfony2 используется для автозагрузки файлов, которые соответствуют либо стандартам технической совместимости с заменяемыми значениями PHP 5.3, либо стандарту именования классов PEAR. Как вы можете видеть, все зависимости хранятся в папке vendor/, но это лишь условное соглашение. Вы можете хранить их где угодно, глобально — на сервере или локально — в своих проектах.

4.2. Система связок
Этот раздел начинается с изучения составляющих одной из главных и наиболее мощных особенностей Symfony2 — системы связок.

Связка — нечто вроде плугина в другом софте. Тогда почему это называется связкой, а не плагином? Потому что всё в Symfony2, от ядра фреймворка до кода, является связкой. Связки — граждане высшего ранга в стране Symfony2. Это обеспечивает гибкость в использовании пакетов встроенных особенностей в сторонних связках или позволяет создавать свои собственные связки. И становится легче легкого выбрать необходимую фичу для своего приложения и оптимизировать ее нужным вам образом.

Приложение состоит из связок, как определено в методе registerBundles() класса AppKernel:

  1. // app/AppKernel.php
  2. public function registerBundles()
  3. {
  4.     $bundles = array(
  5.         new Symfony\Bundle\FrameworkBundle\FrameworkBundle(),
  6.         new Symfony\Bundle\TwigBundle\TwigBundle(),
  7.  
  8.         // enable third-party bundles
  9.         new Symfony\Bundle\ZendBundle\ZendBundle(),
  10.         new Symfony\Bundle\SwiftmailerBundle\SwiftmailerBundle(),
  11.         new Symfony\Bundle\DoctrineBundle\DoctrineBundle(),
  12.         //new Symfony\Bundle\DoctrineMigrationsBundle\DoctrineMigrationsBundle(),
  13.         //new Symfony\Bundle\DoctrineMongoDBBundle\DoctrineMongoDBBundle(),
  14.  
  15.         // register your bundles
  16.         new Application\HelloBundle\HelloBundle(),
  17.     );
  18.  
  19.     if ($this->isDebug()) {
  20.         $bundles[] = new Symfony\Bundle\WebProfilerBundle\WebProfilerBundle();
  21.     }
  22.  
  23.     return $bundles;
  24. }


Наряду с HelloBundle, о котором уже шла речь раньше, обратите внимание, что ядро также позволяет использовать FrameworkBundle,DoctrineBundle,SwiftmailerBundle и ZendBundle. Все они — часть ядра фремворка.
Каждая связка может быть настроена индивидуально в соответствии с файлами настройки, записанными в формате YAML или XML или PHP. Посмотрите на конфигурацию по умолчанию:

  1. # app/config/config.yml
  2. app.config:
  3.     charset:       UTF-8
  4.     error_handler: null
  5.     csrf_secret:   xxxxxxxxxx
  6.     router:        { resource: "%kernel.root_dir%/config/routing.yml" }
  7.     validation:    { enabled: true, annotations: true }
  8.     templating:
  9.         #assets_version: SomeVersionScheme
  10.     session:
  11.         default_locale: en
  12.         lifetime: 3600
  13.  
  14. ## Twig Configuration
  15. #twig.config:
  16. #    auto_reload: true
  17.  
  18. ## Doctrine Configuration
  19. #doctrine.dbal:
  20. #    dbname:   xxxxxxxx
  21. #    user:     xxxxxxxx
  22. #    password: ~
  23. #doctrine.orm: ~
  24.  
  25. ## Swiftmailer Configuration
  26. #swiftmailer.config:
  27. #    transport:  smtp
  28. #    encryption: ssl
  29. #    auth_mode:  login
  30. #    host:       smtp.gmail.com
  31. #    username:   xxxxxxxx
  32. #    password:   xxxxxxxx


Каждая запись (как app.config) определяет конфигурацию связки.
Каждый environment может переписать конфигурацию по умолчанию, создавая свой специальный файл настроек:

  1. # app/config/config_dev.yml
  2. imports:
  3.     - { resource: config.yml }
  4.  
  5. app.config:
  6.     router:   { resource: "%kernel.root_dir%/config/routing_dev.yml" }
  7.     profiler: { only_exceptions: false }
  8.  
  9. webprofiler.config:
  10.     toolbar: true
  11.     intercept_redirects: true
  12.  
  13. zend.config:
  14.     logger:
  15.         priority: debug
  16.         path:     %kernel.logs_dir%/%kernel.environment%.log


Как мы видели в предыдущей части, приложение состоит из связок, что определено в методе registerBundles(), но откуда же Symfony2 знает, где искать эти связки?

Symfony2 весьма гибка в этом отношении. Метод registerBundleDirs() должен возвращать ассоциативный массив, который отображает заменяемые значения для любого допустимого каталога (локального или глобального):

  1. public function registerBundleDirs()
  2. {
  3.     return array(
  4.         'Application'     => __DIR__.'/../src/Application',
  5.         'Bundle'          => __DIR__.'/../src/Bundle',
  6.         'Symfony\\Bundle' => __DIR__.'/../src/vendor/symfony/src/Symfony/Bundle',
  7.     );
  8. }


Так что, когда вы ссылаетесь на HelloBundle в названии контроллера или шаблона, Symfony2 будет искать его в указанных папках.

Теперь вы понимаете, почему Symfony2 такая гибкая? Распределяйте свои связки между приложениями, храните их локально или глобально, все зависит лишь от вас.

4.3. Использование вендоров
Вероятно, ваше приложение будет зависеть от сторонних библиотек. Они должны храниться в папке src/vendor/. Там уже содержатся библиотеки Symfony2, библиотеки SwiftMailer, Doctrine ORM, система шаблонов Twig и подборка из классов Zend Framework.

4.4. Кэш и логи
Symfony2, возможно, один из самых быстрых фреймворков с полным стеком из всех существующих. Но как можно быть таким быстрым, если парсить и интерпретировать десятки YAML и XML файлов для каждого запроса? Это возможно частично благодаря кэш-системе. Настройки приложения парсятся лишь при первом запросе, а затем компилируются в чистый PHP код, хранящийся в папке cashe/. В среде разработки Symfony2 достаточно умен для очистки кэша, когда мы меняете файл. Но в продашене вам придется самому чистить кэш, когда вы обновляете код или меняете настройки конфигурации.

В разработке приложения нередко возникают разнообразные трудности. Файл логов, хранящийся в папке logs/, расскажет вам все о запросах и поможет исправить проблему как можно быстрее.

4.5.Интерфейс командной строки
Каждое приложение поставляется с консолью, которая помогает его поддерживать. Это позволяет увеличивать продуктивность путем автоматизирования скучных и повторяющихся заданий.
Запускайте консоль без аргументов, чтобы узнать больше о ее возможностях:

  1. $ php app/console


Опция --help поможет вам найти и использовать необходимую команду:

  1. $ php app/console router:debug --help


4.6. Заключительные слова
Можете назвать меня безумцем, но после прочтения этой части вам, должно быть, станет проще настраивать Symfony2 и заставить ее работать нужным вам способом. Все в Symfony2 направлено на достижение вашего удобства. Так что не стесняйтесь переименовывать и перемещать каталоги так, как вам хочется.
